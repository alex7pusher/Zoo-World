# Zoo World Test task. 

## Архитектура и структура

Рекомендую смотреть код с точки входа ZooWorldEntryPoint.

Основной код лежит в `Assets/_Project/Scripts/Runtime`.

`Configs` содержит ScriptableObject-конфиги животных, спавна, движения, мира, пула и UI-эффектов.

`Gameplay/Animals` отвечает за создание животных, пул объектов, контроллер животного и view-компонент на prefab.

`Gameplay/Movement` содержит разные типы движения. Движение задается через `AnimalMovementConfig`, поэтому новый тип движения можно добавить отдельным config и классом движения.

`Gameplay/FoodChain` отвечает за обработку столкновений и правила пищевой цепочки. Логика вынесена в стратегии, чтобы не собирать все условия в одном большом классе.

`Gameplay/Spawning` отвечает за периодический спавн животных и хранение активных животных.

`Gameplay/Statistics` считает статистику по погибшим животным. Счетчики хранятся по `AnimalChainType`, поэтому UI не завязан на фиксированные поля вроде "жертвы" и "хищники".

`UI` содержит UI статистики, сервис перевода world position в screen position и эффект `TastyLabel`.

`Infrastructure/Bootstrap` содержит bootstrap stages, которые подготавливают сервисы и запускают симуляцию.

`Infrastructure/Installers` содержит Zenject installers.

`Infrastructure/Events` содержит общую шину событий и события проекта.

`Utils` содержит небольшие общие утилиты и константы проекта.

Также в `Assets/_Project/Modules` лежат мои кастомные модули, кочующие из проекта в проект.

## Что стоит отметить

Я осознанно нарушал YAGNI для демонстрации расширяемости проекта (например id животных для аналитических ивентов в будущем и тд.).

Конфиги животных вынесены в `ScriptableObject`. В `AnimalConfiguration` задается ID, тип в пищевой цепочке, префаб и движение.

Движение животных расширяется через отдельные `AnimalMovementConfig`, без свитча по типам движения.

Правила пищевой цепочки вынесены в отдельные стратегии. Это проще поддерживать, чем один большой блок условий.

UI построен на моем MVP модуле

UI статистики строится по набору счетчиков, а не по двум жестко заданным полям. Если появится новый тип животного, статистику не придется переписывать с нуля.

`TastyLabel` создается из prefab и пулится. UI не решает сам, когда показывать эффект.

Проект использует Zenject для связывания сервисов, UniTask для асинхронных процессов и UniRx для событий и реактивных значений, DoTween, а также мои модули
